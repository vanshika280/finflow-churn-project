Project summary:
FinFlow is a neobank with 2 million users and quarterly customer churn. The project uses a 1000-record synthetic customer dataset to predict churn probability, estimate customer value, and produce a prioritized retention list. The key idea is to not only predict who may churn, but also identify which customers are most valuable to save first.

ROLE 1: ML ENGINEER

The art of the ML Engineer in this project:
The ML Engineer turns raw customer behavior data into a reliable predictive system. Their job is not only to train a model, but to make sure the model is trained correctly, evaluated fairly, and converted into useful outputs such as churn probabilities and feature importance.

Basic responsibilities:
- Prepare model features.
- Exclude non-predictive identifiers such as Customer_ID.
- Use Churned only as the target variable.
- Split the dataset into train and test sets.
- Train the required models from scratch.
- Evaluate model performance using suitable metrics.
- Select the best model.
- Save the model and metadata.
- Generate churn probability scores for all customers.

Fundamental concept: What is churn prediction?
Churn prediction is a binary classification problem. The model learns patterns from historical customers and predicts whether a customer belongs to class 0 or class 1.

Class 0:
The customer did not churn.

Class 1:
The customer churned.

In this project, the output is not only a class label. The more useful output is churn probability. For example, a customer with probability 0.82 is more urgent than a customer with probability 0.41.

Why Customer_ID is not used as a feature:
Customer_ID is only an identifier. It does not contain behavioral meaning. If we include it, the model may learn meaningless patterns from ID ordering. This can create noise or false patterns. A model should learn from features like login activity, transaction frequency, support tickets, and feature usage.

Why Churned is not used as a feature:
Churned is the answer we are trying to predict. If we include it as an input feature, it causes target leakage. The model would appear extremely accurate, but only because it was given the answer during training.

Feature preparation:
The ML Engineer used numeric customer behavior features:
- Age
- Account_Age_Months
- Avg_Monthly_Transactions
- Last_Login_Days_Ago
- Support_Tickets
- Feature_Usage_Score
- Avg_Monthly_Revenue

Missing values:
Missing values can break many ML algorithms. In this project, numeric missing values are handled using median imputation. Median is useful because it is less affected by outliers than mean.

Train-test split:
The data is split into training and testing portions:
- 80% training
- 20% testing

The training set is used to teach the model.
The testing set is used to evaluate how the model performs on unseen data.

Why random_state=42:
Random splitting can produce slightly different results each time. random_state=42 makes the split reproducible. This means the same code gives the same result during demo, review, and viva.

Why stratify=y:
Churn data is usually imbalanced. If the split is random without stratification, one set may get too many churned or non-churned customers. Stratification preserves the class ratio in both train and test sets.

Decision Tree Classifier:
A Decision Tree splits the data using rules. For example:
- If Last_Login_Days_Ago > 60, churn risk may increase.
- If Support_Tickets > 5, churn risk may increase.

Why Decision Tree is useful:
- Easy to explain.
- Good baseline model.
- Gives feature importance.
- Business stakeholders can understand decision rules.

Hyperparameters used:
DecisionTreeClassifier(
    max_depth=4,
    min_samples_leaf=20,
    class_weight="balanced",
    random_state=42
)

max_depth=4:
Limits how deep the tree can grow. This reduces overfitting and keeps the model interpretable.

min_samples_leaf=20:
Each final leaf must have at least 20 samples. This avoids rules that are based on very few customers.

class_weight="balanced":
Gives more importance to the minority class, which is usually churned customers. This helps the model pay attention to churn cases.

Gradient Boosting Classifier:
Gradient Boosting builds many weak decision trees one after another. Each new tree tries to correct the mistakes of the previous trees.

Why Gradient Boosting is useful:
- Captures non-linear relationships.
- Usually performs better than a single tree.
- Works well on tabular data.
- Still gives feature importance.

Hyperparameters used:
GradientBoostingClassifier(
    n_estimators=100,
    learning_rate=0.1,
    max_depth=3,
    random_state=42
)

n_estimators=100:
The model builds 100 trees.

learning_rate=0.1:
Controls how strongly each tree contributes. Smaller learning rates make learning more gradual.

max_depth=3:
Each tree is shallow. Shallow trees reduce overfitting and help generalization.

Evaluation metrics:
Accuracy:
Percentage of total correct predictions.

Precision:
Out of customers predicted as churned, how many actually churned.
High precision means fewer false alarms.

Recall:
Out of actual churned customers, how many were detected.
High recall means fewer missed churners.

F1-score:
Harmonic mean of precision and recall. Useful when we need balance.

ROC-AUC:
Measures how well the model separates churners from non-churners across thresholds.

Confusion matrix:
Shows:
- True negatives
- False positives
- False negatives
- True positives

Why accuracy alone is not enough:
Suppose only 8% of customers churn. A model that predicts "no churn" for everyone can get 92% accuracy but is useless. That is why recall, F1-score, and ROC-AUC are more meaningful in churn problems.

Model selection:
Gradient Boosting was selected because it had the stronger F1-score and ROC-AUC:
- F1-score: 0.7042
- ROC-AUC: 0.9443

Feature importance:
Feature importance explains which variables contributed most to model decisions.

Top drivers:
- Last_Login_Days_Ago
- Avg_Monthly_Transactions
- Support_Tickets

Interpretation:
Customers who have not logged in recently, transact less, or raise more support tickets are more likely to churn.

Possible viva questions for ML Engineer:

Question: Why did you use two models?
Answer: The Decision Tree provides an interpretable baseline. Gradient Boosting provides stronger predictive power by combining many small trees. Comparing both helps us choose a model based on evidence, not assumption.

Question: Why did Gradient Boosting perform better?
Answer: Churn behavior can be non-linear. For example, inactivity may be risky only when combined with low transaction activity or high support tickets. Gradient Boosting captures such interactions better than a single tree.

Question: What is overfitting?
Answer: Overfitting happens when a model learns the training data too closely, including noise. It performs well on training data but poorly on new data.

Question: How did you reduce overfitting?
Answer: We limited tree depth, used min_samples_leaf in Decision Tree, used shallow trees in Gradient Boosting, and evaluated on a separate test set.

Question: What is target leakage?
Answer: Target leakage happens when information that would not be available at prediction time is included as a feature. Including Churned as an input would be leakage.

Question: Why predict probability instead of only class?
Answer: Probability supports prioritization. A customer with 0.90 churn probability should be handled before a customer with 0.52, especially if CLV is high.

Question: What threshold did you use?
Answer: For predicted churn, the default threshold is 0.5. For business risk levels, we used Low below 0.35, Medium from 0.35 to 0.65, and High above 0.65.

Question: Can the threshold be changed?
Answer: Yes. If FinFlow wants to catch more churners, it can lower the threshold to increase recall. If it wants fewer false positives, it can raise the threshold to increase precision.

Question: What would you improve with more time?
Answer: Hyperparameter tuning, cross-validation, calibration of probabilities, cost-sensitive learning, campaign response tracking, and model monitoring after deployment.

The art of the Data Analyst in this project:
The Data Analyst discovers what the dataset is saying before modeling. Their role is to validate the data, summarize patterns, create visual evidence, and translate numbers into business observations.

Basic responsibilities:
- Load and inspect the dataset.
- Check shape and columns.
- Check missing values.

Fundamental concept: What is EDA?
EDA means Exploratory Data Analysis. It is the process of understanding the dataset before building a model.

EDA answers:
- How many rows and columns are present?
- Are there missing values?
- Is the target balanced or imbalanced?
- What is the range of each feature?
- Which features look different between churned and non-churned customers?
- Are there possible outliers?
- Are there suspicious values?

Dataset shape:
The dataset has 1000 records and 9 columns after column normalization.

Column normalization:
The original file had:
- Support_Tickets_Last_6M
- Avg_Monthly_Revenue_INR

The project maps them to:
- Support_Tickets
- Avg_Monthly_Revenue

Why this matters:
Real datasets often have slightly different column names. A robust project should handle reasonable variations without failing.

Missing value check:
Missing values must be checked before modeling because missing data can affect statistics, charts, and model training.

Class distribution:
The dataset has:
- 820 non-churned customers
- 180 churned customers

Churn rate:
180 / 1000 = 18%

Why class distribution matters:
If the target is imbalanced, accuracy can be misleading. A data analyst must warn the ML engineer and business team that recall and F1-score are important.

Basic statistics:
The analyst looks at:
- Mean
- Standard deviation
- Minimum
- Maximum
- Median
- Quartiles

These help understand typical customer behavior.

Churn vs non-churn comparison:
The analyst compares average values by target class.

Important observations from this dataset:
- Churned customers had higher Last_Login_Days_Ago.
- Churned customers had lower Avg_Monthly_Transactions.
- Churned customers had more Support_Tickets.
- Churned customers had lower Feature_Usage_Score.

Business interpretation:
Higher Last_Login_Days_Ago:
Customers who have not logged in recently are disengaged.

Lower Avg_Monthly_Transactions:
Customers who transact less may no longer use FinFlow as their primary bank.

Higher Support_Tickets:
More support tickets may indicate friction, dissatisfaction, or unresolved issues.

Lower Feature_Usage_Score:
Customers who do not use many features may not see enough value in the product.

Why histograms are used:
Histograms show how values are distributed. They help compare churned and non-churned customer behavior.

Why boxplots are used:
Boxplots show median, spread, and outliers. They are useful for comparing numeric features across churn classes.

Why correlation heatmap is used:
It shows relationships between numeric variables. It can reveal whether features move together.

Potential data quality checks:
- Are ages realistic?
- Are negative transaction counts present?
- Are support tickets within expected range?
- Is revenue non-negative?
- Are churn labels only 0 or 1?

Possible viva questions for Data Analyst:

Question: Why is EDA important before modeling?
Answer: EDA helps detect data issues, understand target distribution, identify useful patterns, and guide feature selection.

Question: What did the EDA reveal?
Answer: Inactivity, low transactions, high support tickets, and low feature usage were associated with churn.

Question: What does churn rate mean?
Answer: Churn rate is the percentage of customers who left. In this dataset, it is 18%.

Question: What if the dataset has missing values?
Answer: We report them first. For numeric model features, we can impute missing values using median. For business reporting, we should also mention the missing rate.

Question: What if one feature has outliers?
Answer: We inspect it with boxplots and summary statistics. Depending on severity, we may cap, transform, or leave it if tree models can handle it.

Question: Why use visualizations?
Answer: Charts make patterns easier to understand and help business stakeholders trust the analysis.

Question: What is the difference between correlation and causation?
Answer: Correlation means two variables move together. Causation means one directly causes the other. EDA can suggest relationships but does not prove causation.


ROLE 3: BUSINESS ANALYST

The art of the Business Analyst in this project:
The Business Analyst converts model output into business action. They make sure the solution is not just technically correct but financially and operationally useful.

Basic responsibilities:
- Understand FinFlow's churn problem.
- Define why churn matters financially.
- Convert churn probabilities into customer risk levels.
- Define CLV logic.
- Define priority score.
- Create customer segments.
- Design retention strategies.
- Explain expected business value.

Business problem:
FinFlow is losing customers every quarter. For a neobank, churn can reduce:
- Deposits
- Monthly transactions
- Card usage
- Fee income
- Cross-sell potential
- Long-term customer relationship value

Why prediction alone is not enough:
A churn model can say who is likely to leave. But the business also needs to know who is worth saving first.

Example:
Customer A:
- Churn probability = 0.90
- CLV = INR 500

Customer B:
- Churn probability = 0.80
- CLV = INR 5000

Customer B may be a better campaign priority even though Customer A has slightly higher churn probability.

Risk levels:
- Low Risk: Churn_Probability < 0.35
- Medium Risk: 0.35 to 0.65
- High Risk: > 0.65

Why risk levels are useful:
Marketing and support teams may not use raw probabilities every day. Risk labels make the output easier to act on.

CLV formula:
CLV = Avg_Monthly_Revenue x Expected_Months

Expected months:
Expected_Months = max(1, 12 - Last_Login_Days_Ago / 30)

Business logic behind Expected_Months:
Recently active customers are expected to remain longer. Customers inactive for many days are expected to have shorter remaining lifetime. The max(1, ...) prevents CLV from becoming zero or negative.

Why this is a simple CLV:
This is a practical assessment-level CLV. A production CLV model may include:
- Gross margin
- Discount rate
- Product holding
- Future revenue prediction
- Campaign cost
- Retention probability
- Customer acquisition cost

Priority score:
Priority_Score = Churn_Probability x CLV

Meaning:
Priority score is expected value at risk. A high score means the customer is both likely to churn and valuable.

Customer segments:
High Priority:
Churn_Probability >= 0.65 and CLV in top 33%.

Medium Priority:
Churn_Probability >= 0.35 and not High Priority.

Low Priority:
All others.

Why top 33% CLV:
This creates a practical high-value group. It prevents the campaign from spending too much on low-value customers.

Current segment results:
- High Priority: 32 customers
- Medium Priority: 158 customers
- Low Priority: 810 customers

Top-20 save list:
The top-20 list is sorted by Priority_Score descending.

Top-20 CLV:
INR 79,410.05

Why top-20 matters:
It gives the marketing team a manageable first campaign list. Instead of looking at all customers, they can start with the highest expected value at risk.

Retention strategy 1:
High-value inactive customer reactivation.

Actions:
- Personalized in-app message.
- Email and push notifications.
- Cashback on next transaction.
- Reminder about bill pay, savings pockets, budgeting, card controls.
- Track login and transaction after campaign.

Business logic:
Inactive high-value customers may still be recoverable. A transaction-linked incentive can restart the usage habit.

Retention strategy 2:
Service recovery for friction-heavy customers.

Actions:
- Priority callback.
- Faster complaint resolution.
- Fee waiver where appropriate.
- Relationship manager outreach.
- Post-resolution satisfaction check.

Business logic:
High support tickets indicate friction. Fast service recovery can protect trust and prevent account closure.

Possible viva questions for Business Analyst:

Question: Why did you multiply churn probability by CLV?
Answer: This estimates the value at risk. It prioritizes customers who are both likely to churn and financially valuable.

Question: Why not target all high-risk customers?
Answer: Campaign budgets and support capacity are limited. Prioritization helps allocate resources where the expected business impact is highest.

Question: Why use Avg_Monthly_Revenue for CLV?
Answer: Revenue is a direct measure of customer value. Multiplying it by expected months gives a simple estimate of future value.

Question: What if revenue is not available?
Answer: Use Avg_Monthly_Transactions as a proxy, but clearly state that true CLV needs revenue or margin data.

Question: How would you measure campaign success?
Answer: Compare treated and control groups. Track retention after 30/60/90 days, login activity, transaction activity, feature usage, and revenue retained.

Question: What is the business risk of false positives?
Answer: False positives are customers predicted to churn but who would not churn. They may receive unnecessary incentives, increasing campaign cost.

Question: What is the business risk of false negatives?
Answer: False negatives are actual churners missed by the model. The business loses customers it could have saved.

Question: Which is worse in churn?
Answer: Usually false negatives are more costly because missed churners leave. But if retention incentives are expensive, false positives also matter.
